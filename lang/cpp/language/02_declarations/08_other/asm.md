# asm 声明 - 内联汇编

## 1. 概述

`asm` 声明（asm-declaration）提供了在 C++ 程序中嵌入汇编语言源代码的能力。这是一种**条件性支持**（conditionally-supported）且**实现定义**（implementation-defined）的特性，意味着：

- 编译器可能不支持此特性
- 即使编译器支持，其具体行为也因编译器而异，没有统一的标准语义

内联汇编主要用于需要直接控制硬件、进行极致性能优化或与底层系统交互的场景。

## 2. 来源与演变

### 历史背景

内联汇编的概念源于 C 语言，早期的系统编程需要直接嵌入汇编指令来：
- 访问特殊硬件寄存器
- 执行 CPU 特定指令
- 实现高性能关键路径

### C++98 标准

`asm` 声明从 C++98 开始就被纳入标准，但标准仅规定其为条件性支持，未定义具体语义。各编译器厂商自行实现：

- **GCC**：采用扩展汇编语法（Extended Assembly）
- **MSVC**：采用 `__asm` 块语法
- **Intel C++**：支持 GCC 风格的扩展汇编

### C++11 变化

- 新增**属性（attributes）**支持：可在 `asm` 声明前添加属性
- 明确规定 `asm` 声明为**实现定义**行为

### C++20 变化

- 允许在 `constexpr` 函数中使用 `asm` 声明（通过 `__cpp_constexpr` 特性测试宏，值为 `201907L`）
- 支持平凡默认初始化和 `asm` 声明

### C++26 变化

- 语法扩展：从 `string-literal`（字符串字面量）改为 `balanced-token-seq`（平衡标记序列）
- 这允许更灵活的内联汇编语法，如 GCC 扩展汇编语法

### 缺陷报告

| 缺陷报告 | 应用版本 | 原行为 | 修正行为 |
|---------|---------|--------|---------|
| CWG 195 | C++98 | 要求支持所有 asm 声明 | 改为条件性支持 |
| CWG 2262 | C++11 | 属性不能应用于 asm 声明 | 允许应用属性 |

## 3. 语法与参数

### 基本语法

**C++26 之前**：

```cpp
attr(可选) asm ( string-literal ) ;
```

**C++26 起**：

```cpp
attr(可选) asm ( balanced-token-seq ) ;
```

### 参数说明

| 参数 | 说明 |
|------|------|
| `attr` | （C++11 起）任意数量的属性 |
| `string-literal` | 字符串字面量，与普通字符串字面量规则相同，包括原始字符串字面量 |
| `balanced-token-seq` | 平衡标记序列，其中圆括号、方括号和花括号必须平衡；具体限制和含义由实现定义 |

### 语法要点

1. **字符串字面量**：通常是一个用汇编语言编写的短程序
2. **平衡标记序列**：允许更复杂的语法结构，如 GCC 扩展汇编的输入/输出操作数
3. **执行时机**：每当执行到此声明时，汇编代码被执行

### 声明位置

`asm` 声明可以出现在：
- **块内**（block）：函数体内或其他复合语句中
- **块外**：与其他声明一样，可出现在全局作用域

## 4. 底层原理

### 编译器处理机制

`asm` 声明的处理完全依赖于编译器实现：

```
源代码 (C++)
    |
    v
编译器前端 (解析 asm 声明)
    |
    v
编译器后端 (将汇编代码传递给汇编器)
    |
    v
汇编器 (生成目标代码)
    |
    v
链接器 (生成可执行文件)
```

### 编译器差异

不同编译器对内联汇编的支持存在巨大差异：

| 编译器 | 语法风格 | 特点 |
|--------|---------|------|
| GCC/Clang | AT&T 语法 + 扩展汇编 | 支持操作数约束、寄存器分配 |
| MSVC | Intel 语法 + `__asm` 块 | 更接近 C 风格，但不支持 x64 |
| Intel C++ | 两者皆可 | 兼容 GCC 扩展汇编 |

### GCC 扩展汇编语法

GCC 的扩展汇编是最广泛使用的形式，现已被 Intel、IBM、Sun 等编译器支持：

```cpp
asm ( "汇编模板"
      : 输出操作数   // 可选
      : 输入操作数   // 可选
      : 被破坏的寄存器列表  // 可选
);
```

**操作数约束**：
- `"r"` - 任意通用寄存器
- `"m"` - 内存操作数
- `"="` - 只写输出
- `"+"` - 读写操作数

### 调用约定

内联汇编需要了解平台的调用约定（calling convention），例如：
- **x86-64 Linux**：参数通过 RDI、RSI、RDX、RCX、R8、R9 传递，返回值在 RAX
- **x86-64 Windows**：参数通过 RCX、RDX、R8、R9 传递

### 性能特征

| 方面 | 说明 |
|------|------|
| 编译时开销 | 几乎无，汇编代码直接传递给汇编器 |
| 运行时性能 | 最优，直接使用底层指令 |
| 可移植性 | 极差，依赖特定平台和编译器 |
| 可读性 | 较差，需要汇编知识 |

## 5. 使用场景

### 适用场景

| 场景 | 说明 |
|------|------|
| 硬件访问 | 访问特殊 CPU 寄存器、执行特权指令 |
| 性能优化 | 编译器无法生成的最优指令序列 |
| 系统调用 | 直接执行操作系统系统调用 |
| 嵌入式开发 | 与特定硬件外设交互 |
| 安全编程 | 侧信道防护、密码学实现 |

### 不适用场景

| 场景 | 推荐替代 | 原因 |
|------|---------|------|
| 一般应用开发 | 内置函数（intrinsics） | 可移植性差 |
| 跨平台代码 | 标准库或第三方库 | 平台依赖 |
| 初学者项目 | 无 | 难以调试和维护 |

### 最佳实践

1. **优先使用编译器内置函数**：如 `__builtin_ctz`、`_mm_add_ps` 等，比内联汇编更具可移植性
2. **隔离平台相关代码**：使用条件编译或分离到不同源文件
3. **添加详细注释**：解释汇编代码的功能和约束
4. **验证输入输出**：确保与 C++ 代码的数据交互正确
5. **声明破坏列表**：正确列出被修改的寄存器

### 注意事项

1. **编译器差异**：GCC、Clang、MSVC 的语法完全不同
2. **平台差异**：x86、ARM、RISC-V 等架构的汇编指令不同
3. **寄存器约定**：必须遵守平台的 ABI（Application Binary Interface）
4. **优化干扰**：编译器可能不知道汇编代码的副作用
5. **调试困难**：内联汇编代码难以调试

### 常见陷阱

1. **忘记声明破坏的寄存器**：导致编译器错误优化
2. **错误的约束字符串**：导致寄存器分配错误
3. **忽视内存屏障**：多线程环境下可能导致数据竞争

## 6. 代码示例

### 基础用法：定义汇编函数

以下示例演示 GCC/Clang 编译器下的两种内联汇编语法（x86-64 Linux 平台）：

```cpp
#include <iostream>

// 声明一个用汇编实现的外部函数
extern "C" int func(int x);

// 使用 asm 声明定义函数体
// 原始字符串字面量在这里非常有用
asm(R"(
.globl func
    .type func, @function
    func:
    .cfi_startproc
    movl %edi, %eax  /* x86-64 调用约定：参数在 RDI */
    addl $1, %eax    /* 返回 x + 1 */
    ret
    .cfi_endproc
)");

int main() {
    int n = func(0110);  // 八进制 0110 = 十进制 72
    std::cout << "func(72) = " << n << std::endl;  // 输出 73
    return 0;
}
```

### 高级用法：GCC 扩展汇编

使用 GCC 扩展汇编语法进行计算，展示了 P2361R6 提案后符合标准的语法：

```cpp
#include <iostream>

int main() {
    int n = 73;

    // GCC 扩展汇编：计算 n * 5
    // "leal (%0,%0,4),%0" 是 lea 指令，计算 %0 + %0*4 = %0*5
    asm ("leal (%0,%0,4),%0"
         : "=r" (n)      // 输出：任意寄存器，写入 n
         : "0" (n)       // 输入：使用与输出相同的寄存器
    );

    std::cout << "73 * 5 = " << n << std::endl;  // 输出 365
    return 0;
}
```

### 系统调用示例

直接使用内联汇编执行 Linux 系统调用：

```cpp
#include <iostream>

int main() {
    int n = 73;

    // GCC 扩展汇编：计算 n * 5
    asm ("leal (%0,%0,4),%0"
         : "=r" (n)
         : "0" (n));

    std::cout << "73*5 = " << n << std::endl;  // flush is intentional

    // 标准内联汇编：执行 Linux exit 系统调用
    // 注意：这会直接退出程序
    asm ("movq $60, %rax\n"   // exit 系统调用号
         "movq $2,   %rdi\n"  // 退出码 2
         "syscall");

    // 不会执行到这里
    return 0;
}
```

**程序输出**：
```
73*5 = 365
```
程序随后以退出码 2 终止。

### 常见错误及修正

#### 错误 1：忘记声明破坏的寄存器

```cpp
// 错误：修改了 eax 但未告知编译器
int x = 10, y;
asm ("movl $5, %eax");
// 编译器可能仍认为 eax 保存着其他值

// 修正：声明破坏列表
int x = 10, y;
asm ("movl $5, %eax" : : : "eax");  // 正确声明破坏的寄存器
```

#### 错误 2：错误的操作数约束

```cpp
// 错误：使用只读约束 "r" 作为输出
int value;
asm ("movl $42, %0" : "r" (value));  // 错误！"r" 是输入约束

// 修正：使用输出约束 "="
int value;
asm ("movl $42, %0" : "=r" (value));  // 正确： "=r" 是只写输出约束
```

#### 错误 3：忽视内存屏障

```cpp
// 错误：多线程环境下可能导致数据竞争
int flag = 0;
asm ("movl $1, %0" : "=m" (flag));  // 编译器可能重排内存访问

// 修正：添加内存破坏声明
int flag = 0;
asm ("movl $1, %0" : "=m" (flag) : : "memory");  // 正确：声明内存修改
```

#### 错误 4：跨平台编译问题

```cpp
// 错误：假设特定平台寄存器
void example() {
#ifdef _WIN32
    // MSVC 不支持 GCC 扩展汇编语法
    asm ("movl $1, %eax");  // 错误！MSVC 不认识这种语法
#endif
}

// 修正：使用条件编译或内置函数
void example() {
#if defined(__GNUC__) || defined(__clang__)
    asm ("movl $1, %eax");
#elif defined(_MSC_VER)
    __asm mov eax, 1  // MSVC 语法
#endif
}
```

## 7. 总结

`asm` 声明是 C++ 中一个**条件性支持且实现定义**的特性，用于在 C++ 程序中嵌入汇编代码。

### 核心要点

| 要点 | 说明 |
|------|------|
| 可移植性 | 极差，依赖编译器和平台 |
| 性能 | 最优，直接生成底层指令 |
| 使用难度 | 高，需要汇编知识和 ABI 理解 |
| 标准程度 | 标准只规定语法，语义完全实现定义 |

### 版本演进

- **C++98**：引入基本 `asm` 声明
- **C++11**：支持属性，明确实现定义
- **C++20**：允许在 `constexpr` 函数中使用
- **C++26**：扩展为 `balanced-token-seq`，支持 GCC 扩展汇编语法

### 使用建议

1. **最后手段**：仅在性能关键且其他方法无法达成时使用
2. **优先替代**：优先考虑编译器内置函数（intrinsics）
3. **充分注释**：详细说明汇编代码的功能和约束
4. **隔离代码**：将平台相关代码隔离到单独文件
5. **谨慎调试**：充分测试，内联汇编难以调试

### 相关概念

| 概念 | 关系 |
|------|------|
| 编译器内置函数（intrinsics） | 更可移植的替代方案 |
| 内联函数（inline） | 编译器内联，非汇编嵌入 |
| `extern "C"` | 与汇编函数交互时常用 |
| ABI（Application Binary Interface） | 内联汇编必须遵守的规范 |
