# C++ 属性：assume（C++23 起）

## 1. 概述

`[[assume]]` 是 C++23 标准引入的属性（attribute），用于告知编译器在程序特定点某个表达式总是评估为 `true`。这允许编译器基于该假设信息进行优化，从而生成更高效的代码。

**技术定位**：
- 属于编译器优化提示属性
- 与运行时断言（assert）配合使用
- 标准化了各编译器厂商的假设扩展功能

**主要用途**：
- 向编译器提供程序不变量（invariant）信息
- 启用更激进的代码优化
- 在断言被禁用时提供优化机会

## 2. 来源与演变

### 历史背景

在 C++23 标准化之前，各编译器已提供类似的假设功能：

| 编译器 | 内置语法 |
|--------|----------|
| Clang | `__builtin_assume(condition)` 和 `__attribute__((assume))` |
| GCC | `__attribute__((assume(...)))` |
| MSVC | `__assume(condition)` |

这些编译器扩展语法不统一，代码可移植性差。C++23 的 `[[assume]]` 提供了标准化的语法，使代码更具可移植性。

### 首次引入

`[[assume]]` 属性在 **C++23** 标准中首次引入，标准化编号为 ISO/IEC 14882:2024 第 9.12.3 节 [dcl.attr.assume]。

### 设计动机

1. **统一语法**：提供跨编译器一致的标准语法
2. **优化机会**：允许编译器基于假设进行激进优化
3. **调试兼容**：可与 `assert` 宏配合使用
4. **零运行时开销**：假设表达式不会被实际求值

### 缺陷报告

| DR 编号 | 应用版本 | 已发布行为 | 修正行为 |
|---------|----------|------------|----------|
| CWG 2924 | C++23 | 违反假设会导致未定义行为 | 改为运行时未定义行为（runtime-undefined behavior） |

**重要区别**：运行时未定义行为意味着只有当程序实际执行到假设点且假设为假时才会触发未定义行为，而非无条件触发。

## 3. 语法与参数

### 基本语法

```cpp
[[assume(expression)]]
```

### 参数说明

| 参数 | 说明 |
|------|------|
| `expression` | 任意表达式（除未加括号的逗号表达式外） |

**表达式特性**：
- 表达式会进行上下文到 `bool` 的转换
- 表达式**不会被求值**（not evaluated），但仍被视为潜在求值（potentially evaluated）
- 这意味着表达式中的函数调用不会真正执行

### 语法规则

`[[assume]]` 属性只能应用于空语句（null statement）：

```cpp
[[assume(x > 0)]];  // 正确：应用于空语句
[[assume(x > 0)]]   // 错误：缺少分号，不是空语句
int y [[assume(x > 0)]] = 0;  // 错误：不能应用于声明
```

### 逗号表达式处理

如果需要使用逗号表达式，必须用括号括起来：

```cpp
[[assume((a, b))]];  // 正确：逗号表达式需要括号
// [[assume(a, b)]];  // 错误：未括号化的逗号表达式
```

### 语义说明

- 如果转换后的表达式在假设出现点评估为 `true`，则假设无效果
- 如果转换后的表达式评估为 `false`，则假设的评估具有**运行时未定义行为**（runtime-undefined behavior）

## 4. 底层原理

### 编译器优化机制

当编译器遇到 `[[assume(expr)]]` 时，会进行以下处理：

```
编译器处理流程：
    │
    ├── 1. 静态分析阶段：记录 expr 在该点为真的假设
    │
    ├── 2. 优化阶段：基于假设进行代码变换
    │
    └── 3. 运行时：假设本身不生成任何代码
```

编译器可能进行的优化类型：

| 优化类型 | 说明 |
|----------|------|
| 死代码消除 | 如果某分支条件与假设矛盾，可消除该分支 |
| 常量传播 | 假设变量值为特定值时，可进行常量替换 |
| 循环优化 | 假设循环不变量可启用向量化等优化 |
| 函数内联决策 | 基于假设信息改进内联决策 |

### 与 assert 的对比

| 特性 | `assert` | `[[assume]]` |
|------|----------|--------------|
| 调试时行为 | 检查条件，失败时终止程序 | 无检查 |
| 发布时行为 | `NDEBUG` 定义时无效果 | 提供优化机会 |
| 表达式求值 | 求值表达式 | 不求值表达式 |
| 违反后果 | 运行时断言失败 | 未定义行为 |

### 表达式求值规则

```
表达式不求值但可能潜在求值
    │
    ├── 意味着：表达式中出现的函数/变量必须已声明且可访问
    │
    └── 但意味着：表达式不会被执行
```

### 假设的作用范围

**关键特性**：假设仅在声明点有效，不会向后传播。

```cpp
[[assume(x == 3)]];
h();       // 编译器可能假设 x 仍为 3
g(x);      // 编译器可能替换为 g(3)
h();       // 无法保证 x 的值
g(x);      // 编译器不能替换为 g(3)
```

## 5. 使用场景

### 适用场景

| 场景 | 说明 |
|------|------|
| 与断言配合 | 在 `NDEBUG` 定义时提供优化机会 |
| 优化热路径 | 在关键性能路径上帮助编译器消除分支 |
| 表达程序不变量 | 告知编译器程序员知道为真但编译器无法推导的条件 |
| 函数返回假设 | 假设函数调用会返回（不会抛出异常或终止） |

### 最佳实践

#### 推荐模式：断言 + 假设

这是使用 `[[assume]]` 的推荐模式：

```cpp
#include <cassert>

void process_positive(int value) {
    assert(value > 0);       // 调试版本：运行时检查（NDEBUG 未定义时）
    [[assume(value > 0)]];   // 发布版本：提供优化信息（始终有效）

    // 编译器可安全地进行除法优化
    int result = 100 / value;
}
```

**原因**：
- `assert` 在 `NDEBUG` 未定义时验证假设
- `[[assume]]` 在 `NDEBUG` 定义时提供优化
- 两者配合确保调试和发布版本的一致性

### 注意事项

| 注意点 | 说明 |
|--------|------|
| 谨慎使用 | 假设为假时会导致运行时未定义行为 |
| 确保假设正确 | 错误的假设会产生难以调试的 bug |
| 作用域有限 | 假设仅在声明点有效 |
| 表达式限制 | 不能使用未括号化的逗号表达式 |
| 副作用丢失 | 假设表达式中的函数调用不会执行 |

### 常见陷阱

#### 陷阱 1：假设过期

```cpp
// 错误：变量修改后假设不再有效
void wrong_example(int& x) {
    [[assume(x > 0)]];
    x = -5;  // x 值已改变，之前的假设不再有效！
    // 编译器仍可能基于过期假设优化
}
```

#### 陷阱 2：依赖副作用

```cpp
#include <iostream>

// 错误：假设表达式中的副作用不会执行
void wrong_side_effect(int x) {
    int counter = 0;
    [[assume((counter++, x > 0))]];  // counter++ 不会执行！
    // counter 仍然是 0，不是 1
}
```

#### 陷阱 3：函数调用后假设失效

```cpp
void example(int& x) {
    [[assume(x == 5)]];
    some_function();  // 可能修改 x
    use(x);           // 编译器不能假设 x == 5
}
```

## 6. 代码示例

### 示例 1：基础用法

```cpp
#include <cassert>

void process(int& x)
{
    // 调试时验证，发布时提供优化信息
    assert(x > 0);
    [[assume(x > 0)]];

    // 编译器可能生成更高效的代码
    // 因为它知道 x 是正数
    int result = x / 2;  // 可能优化为移位操作
}
```

### 示例 2：编译器优化示例

```cpp
#include <cmath>

void f(int& x, int y)
{
    void g(int);
    void h();

    // 编译器假设 x 为正数
    [[assume(x > 0)]];

    // 可能生成更高效的代码（如使用无符号除法）
    g(x / 2);

    x = 3;
    int z = x;

    // 编译器假设调用 h() 后 x 值不变
    // 注意：h() 不会被实际调用（表达式不求值）
    [[assume((h(), x == z))]];

    h();
    g(x);  // 编译器可能将其优化为 g(3)

    h();
    g(x);  // 编译器不能将其优化为 g(3)
           // 假设仅在其出现点有效

    z = std::abs(y);

    // 假设 g(z) 会正常返回
    // g(z) 调用不会实际执行
    [[assume((g(z), true))]];

    g(z);  // 结合下面的假设，可能替换为 g(10)

    // 如果 y != -10 则未定义行为
    [[assume(y == -10)]];

    // 编译器可能推断 x = 5
    [[assume((x - 1) * 3 == 12)]];
    g(x);  // 可能优化为 g(5)
}
```

### 示例 3：正确与错误用法对比

```cpp
#include <cassert>

// === 正确用法 ===

// 1. 与断言配合
void correct_example_1(int* ptr)
{
    assert(ptr != nullptr);
    [[assume(ptr != nullptr)]];
    ptr->method();  // 编译器可移除空指针检查
}

// 2. 表达已知不变量
void correct_example_2(int divisor)
{
    // 从业务逻辑知道 divisor 绝不会为 0
    assert(divisor != 0);
    [[assume(divisor != 0)]];
    int result = 100 / divisor;  // 编译器可移除断零检查
}

// === 错误用法 ===

// 1. 假设可能为假的条件
void wrong_example_1(int x)
{
    [[assume(x > 0)]];  // 错误：x 可能 <= 0
    // 如果 x <= 0，程序行为未定义
}

// 2. 忘记分号（不是空语句）
void wrong_example_2(int x)
{
    [[assume(x > 0)]]   // 错误：语法错误，缺少分号
    int y = x;
}

// 3. 假设在后续代码中仍然有效
void wrong_example_3(int& x)
{
    [[assume(x == 5)]];
    x = 10;  // x 被修改
    // 这里假设 x == 5 不再成立
    use(x);  // 编译器可能错误优化
}
```

### 示例 4：使用逗号表达式的正确方式

```cpp
void example_comma(int x)
{
    void g(int);

    // 错误：未括号化的逗号表达式
    // [[assume(g(x), true)]];  // 语法错误

    // 正确：用括号包围逗号表达式
    [[assume((g(x), true))]];  // g(x) 不会被调用，只是假设
}
```

## 7. 总结

### 核心要点

| 要点 | 说明 |
|------|------|
| 标准版本 | C++23 起 |
| 语法形式 | `[[assume(expression)]];` |
| 应用位置 | 只能应用于空语句 |
| 表达式求值 | 不求值（no evaluate） |
| 违反后果 | 运行时未定义行为 |
| 主要用途 | 提供编译器优化机会 |
| 作用范围 | 仅在声明点有效 |

### 与相关概念对比

| 特性 | `[[assume]]` | `assert` | `[[unreachable]]` |
|------|--------------|----------|-------------------|
| 引入版本 | C++23 | C 遗留 | C++23 |
| 调试检查 | 无 | 有 | 无 |
| 发布优化 | 有 | 无 | 有 |
| 语义 | 条件假设 | 条件验证 | 不可达声明 |
| 表达式求值 | 不求值 | 求值 | 无表达式 |

### 学习建议

1. **谨慎使用**：`[[assume]]` 是高级特性，仅在性能关键路径使用
2. **配合断言**：始终与 `assert` 配合，确保调试时能验证假设
3. **文档化**：在代码注释中说明假设的来源和正确性保证
4. **测试充分**：在启用和禁用假设的场景下都要进行充分测试
5. **默认不使用**：仅在性能测试表明需要时才使用
